# Agent Instruction Pack

This folder is an instruction set for AI coding assistants, tailored to work
with the actual codebase in this workspace.

## How To Use This Pack

Start with this file, then load only the supporting docs that match the current
task.

Use this order:

1. `AGENTS.md`
2. `../ai-settings.md` — mode and which modules are active (written by `/tmpl-bootstrap`)
3. `../ai-plugins.json` — plugin manifest, verified by every release command
4. `releases/` — scan for project context (see Requirements below)
5. matching files in `roles/` (only if checked in `ai-settings.md`)
6. matching files in `guides/` (only if checked in `ai-settings.md`)
7. matching files in `guidelines/` (only if checked in `ai-settings.md`)
8. matching files in `refactoring/` (only if checked in `ai-settings.md`)
9. `../docs/` — onboarding, gitflow, code-review, per-role, per-language conventions (produced by `/tmpl-setup`)
10. repo docs such as `README.md` and system/API documentation
11. the actual project code

## Pipeline Configuration

Read `../ai-settings.md` before loading any role, guide, or refactoring doc.

- Items marked `[x]` are active — load them when they match the current task
- Items marked `[ ]` are disabled — skip them even if they match
- `releases/` is always scanned regardless of pipeline configuration

## Requirements

Scan `releases/` (the canonical post-`/tmpl-setup` home) and any other
conventional requirements locations the project uses (`requirements/`,
`docs/requirements/`, `specs/`, top-level `REQUIREMENTS.md` / `PRD.md` /
`SPEC.md`) to understand what the project is and what it should do.

Read everything in the folder recursively — any files, any subfolders, any format.
Requirements may be structured or unstructured, detailed or sparse. Build your
understanding from whatever is there. This is the ground truth for the project's intent.

## Default Mental Model

Before editing, understand how the request fits into the system:

{DESCRIBE_YOUR_PIPELINE_OR_DATA_FLOW_HERE — numbered stages from input to output}

If a change touches the pipeline, reason about the effect on:

- input validation and schema resolution
- core processing stages
- output validation and assembly
- persisted outputs and metadata
- verification and testing flow

## Core Rules

- Read the current code, docs, and configuration before making assumptions.
- Project-local docs and code behavior override stale prompt lore.
- Preserve established architectural boundaries unless the task explicitly changes them.
- Do not invent IDs, geometry, or data that should come from upstream stages.
- Keep processing stage responsibilities separated.
- Prefer clear contracts between stages over implicit coupling.
- For non-trivial changes, make a short file plan before editing.
- Define schemas, models, payload shapes, and storage impacts early.
- For bugs, reproduce the issue or read the failing path first, then fix the root cause.
- Verify with the smallest reliable check first, then broader service-level checks.
- Keep durable rules in files, not in memory or chat history.

## Task Protocol

### Non-trivial work

1. Explore the codebase and identify affected files.
2. Summarize the plan before large edits.
3. Identify models, schemas, contracts, and runtime artifacts that may change.
4. Update or add verification for the changed behavior.
5. Implement in focused steps.
6. Run the most relevant checks.
7. Self-check against the relevant role and technology guide before finishing.

### Bug fixes

1. Reproduce the issue, inspect the failing path, or read the failure artifact.
2. Identify the root cause.
3. Fix the source, not just the symptom.
4. Add or update regression coverage where the repo supports it.
5. Confirm the issue is resolved through the narrowest reliable verification.

### Refactoring

1. Identify all violations of the rule you are enforcing.
2. Batch work by file or pipeline stage.
3. Finish one unit before moving to the next.
4. Re-run verification after each file or small batch.
5. Keep refactors separate from feature work when the scope is broad.
6. Use the docs in `refactoring/` that match the current stack.

## Role Selection

These role files are thinking modes, not separate agents. Pick the role that fits
the work and combine it with the matching guide. Only activate roles that are
checked in `ai-settings.md`.

**Curated catalog** — match `team_roles[].name` from `configure.json` against
this list. Slugs not in the list are generated by `/tmpl-bootstrap` from
`bootstrap/templates/role.template.md`.

- requirements, planning, scope clarification: `roles/business-analyst.md`
- test rigor, verification, regression prevention: `roles/qa-engineer.md`
- backend and API system design: `roles/backend-architect.md`
- backend implementation (any language): `roles/backend-developer.md`
- frontend implementation (any framework): `roles/frontend-developer.md`
- UI and design-system work: `roles/ui-ux-designer.md`
- ML, LLM, extraction, prompting, evaluation: `roles/ml-engineer.md`
- CI, deployment, infrastructure, monitoring: `roles/devops-engineer.md`
- IaC, internal-developer-platform, deployment pipelines: `roles/platform-engineer.md`
- production reliability, oncall, SLO discipline: `roles/sre.md`
- public API design for libraries / packages / SDKs: `roles/library-author.md`
- user-facing documentation, tutorials, references: `roles/tech-writer.md`
- data pipelines, ETL, schema evolution, analytics: `roles/data-engineer.md`
- iOS / Android / React Native / Flutter: `roles/mobile-developer.md`
- threat modelling, secure design, vuln response: `roles/security-engineer.md`

The template makes **no assumption** about which roles a project needs —
the set comes from `configure.json.team_roles[]`, populated during
`/tmpl-setup` from requirements + user dialogue. The exact stack each role
works against is determined by the guides loaded alongside it (see
`ai-plugins.json.plugins.guides[]` and the per-role "stack-guides" markers
in `roles/{role}.md`).

SECURITY/CONTRIBUTING/code-review, 5j per-role docs, 5k per-language

project-specific context that isn't in the role/guide files. The
